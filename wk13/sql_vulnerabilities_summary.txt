1) Input: Ben

Observation:
The page returned only the row where firstname is Ben and active equals 1.

Expected?
Yes, this is expected.

Explanation:
The query worked as intended because the input was treated as a normal string value. It searched for rows where firstname matched Ben and active was 1, so only the active Ben user was returned.


2) Input: Ben'-- 

Observation:
The page returned all rows where firstname is Ben, including the inactive user whose active value is 0.

Expected?
No, this is not expected in a secure application.

Explanation:
The single quote closed the original string in the SQL statement, and the double dash commented out the rest of the query. This removed the condition AND active = 1, so the database returned both active and inactive users named Ben. This demonstrates a SQL injection vulnerability.


3) Input: Ben' or 1=1 

Observation:
The page returned more rows than expected, not just the active Ben user.

Expected?
No, this is not expected in a secure application.

Explanation:
The injected condition or 1=1 changed the logic of the WHERE clause. Because 1=1 is always true, the query matched many rows that should not have been returned. This demonstrates another form of SQL injection vulnerability.
